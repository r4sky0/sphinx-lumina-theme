## Design Context

### Users
- **Primary audience**: Developers of all levels — from open-source maintainers documenting their libraries, to enterprise teams building internal docs, to solo developers shipping personal projects.
- **Reader context**: Mixed — developers scanning API references quickly, and less technical users following guides and tutorials. The theme must serve both efficiently.
- **Emotional goal**: **Confident clarity** — readers should feel oriented and focused. The docs respect their time. Not cold or corporate, but warm enough to feel inviting without being playful. The feeling of a well-organized workspace with good lighting.

### Brand Personality
- **Three words**: Crisp, welcoming, precise
- **Voice**: Professional but approachable. Direct without being terse. Helpful without being patronizing.
- **Tone**: The interface should feel like a product, not a template. Every element earns its place.

### Aesthetic Direction
- **Visual tone**: Modern, clean, editorial. Generous whitespace. Documentation as a first-class product experience.
- **References**: Fumadocs and Nextra — a mix of both. Fluid navigation, polished feel, treats docs as seriously as the software they describe.
- **Anti-references**: Old-school plain Sphinx themes (alabaster, classic). No cramped layouts, no dated typography, no "afterthought" documentation styling.
- **Theme strategy**: Emerald accent is the default theme. The architecture supports future alternative themes via CSS custom properties. Design decisions should remain theme-agnostic where possible.
- **Dark mode**: Full dark mode support, both modes treated as first-class. No "dark mode as afterthought."

### Design Principles
1. **Content is king** — Every design decision should make the content easier to find, read, and understand. Decoration that doesn't serve comprehension gets cut.
2. **Respect the reader's time** — Fast scanning, clear hierarchy, obvious navigation. Readers should never feel lost.
3. **Modern but not trendy** — Clean, contemporary design that won't feel dated in two years. Avoid chasing visual trends (glassmorphism, neon glows, gradient text).
4. **Accessible by default** — WCAG AA compliance as baseline. Reduced motion support, sufficient contrast, keyboard navigation. Accessibility is not an add-on.
5. **Theme-ready architecture** — CSS custom properties for all visual tokens. Design decisions should make it straightforward to create alternative color themes in the future.

### Technical Constraints
- Built on Sphinx's template system (Jinja2)
- Tailwind CSS v4 for utility classes
- Alpine.js for interactive components
- Must override Sphinx's basic theme cleanly without visual artifacts
- Self-hosted fonts only (no external CDN calls)
- pnpm for JS package management, uv for Python
